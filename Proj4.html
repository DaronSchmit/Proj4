Introduction

Decision making requires sound data which often can be obtained from observation. For example, determining when the busy times are at the mall is possible by collecting data on the times of arrivals and departures of people at each door. Decision making about mall staffing levels can be made based on this relatively easily gathered data. However, there are many areas where the information gathering may be very difficult or even impossible. Such situations include "what if" scenarios or planning for certain future situations or events. To assist in these kinds of decisions, computer modeling can be used. Without actually observing a real-world system, software can be written to model the individual components of such a system. Then, the parameters can be changed to see how the system will behave under a variety of situations. The use of computer models can be a cost-effective tool, or in some cases, the only way to gather information about or predict the behavior of a possible system. Using an object oriented language (like Java) together with basic data structures that we have covered so far in class allows is to relatively easily create the discreet event simulation system to model a real-world scenarios. Note that discreet event simulation is one of many approaches to simulating a system, and it is the one that we will use in this project, Discreet event simulation is based on the creation of data objects that represent each of the components in the system. Time is modeled using an agenda (or priority queue) where future "events" are kept for simulation
in chronological order. Only time steps where an event occurs are considered by the simulation program.

Modeling the Grocery Store Checkout

A local grocery store chain is considering moving from employee bagging to customer bagging of groceries at the checkout lane in the hope that they can reduce costs. However, the cost savings in not having to hire baggers may not make up for the additional costs of hiring some new cashiers. Of course, there are other concerns as well, but we will only consider throughput and the customer experience in our model. There is one more thing--do express lanes really help? You know those lanes for 10 items or fewer? We'll try to model that as well--how does the system compare when one or more of the checkout lanes are express lanes compared with having all checkout lanes the same? (Note that the total number of lanes will remain the same regardless of whether express lanes are used or not.)

The Car Wash example that we "acted out" in lecture has many parallels to this project. You may find it helpful to use some of the code for the Car Wash example in your simulation project, and that is O.K. Specifically, you should use the priority queue (PQ.java) as-is. But, be sure to credit any code "borrowed" from the posted lecture examples.

The Environment to be Modeled


We will assume the following adjustable system parameters:

  • A variable number of checkout lanes
  • Possibility to have one or two lanes restricted to 10 items or fewer
  • Average inter-arrival rate of shoppers at the checkout 

We will assume the following constraints:

  • Shoppers will be passive, but they will contain data such as:

      1) time they entered a checkout line
      2) number of items in their cart (determined statistically, see below)

  • Arrival of a shopper at a checkout lane will be provided by an arrival class that will generate arrivals for all the lanes (determined statistically, see below)
  • Checkout time is determined according to number of items in the shopping cart (see below)
  • Customer bagging increases the per-customer checkout time over employee bagging (see formula below)

System Elements

Simulation of the system is achieved by creating classes to model the behavior of each element of the system as well as each activity (event) that occurs in the system. There is NOT an overall controller that calls for actions to happen at particular times except for a main driver loop that runs queued events until time runs out. Each element in the system models its own behavior and interactions with other elements in the system.

For the Grocery Simulation, we have:

Elements and the Classes to Represent Them

* Shoppers (Shopper) - passive and contain data only
* Arrival generator (ShopperMaker)
* Checkout Lanes (Checkers)
* agenda (PQ)

Events (actions)

Each NON-passive element class has an Event object class associated with it, or in the case of the ShopperMaker where there is only one instance, the class can also be the event. The associated event contains the run() method (as discussed in lecture) that simulates the specific behavior of that class.

* CheckerEvent (occurs each time a Shopper finishes checkout and leaves)
* ShopperEvent (one occurs for each Shopper entering the checkout system)
